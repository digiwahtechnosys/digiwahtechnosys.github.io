Section 01
Engagement Objective
ⓘ
This is not a CRM cleanup. It is a full transformation.

The goal is to evolve the existing Zoho setup from a basic deal-tracking tool into a governed, automated Revenue Management System that aligns with SaaS best practices.

Current State Problems

  • Duplicate work across CRM and Books
  • No structured SaaS lifecycle — especially renewals
  • Weak approval and governance controls
  • Renewal pipeline managed separately — fragmented view
  • No ownership clarity for renewals
  • No ARR / MRR visibility
  • Reporting depends entirely on manual discipline

Target State Goals

  • CRM as single source of truth for all Sales motions
  • Unified Opportunity model: New + Renewal + Expansion
  • Automated renewal lifecycle with alerts and auto-creation
  • Multi-level deal desk approval with per-component thresholds
  • System-calculated revenue metrics (MRR / ARR)
  • Real-time dashboards — Sales, Renewal, Finance, Partner
  • Scalable partner ecosystem with portal access

Section 03
CRM Structure & Module Design

Opportunities Module (Renamed from Deals)

The Deals module will be renamed to Opportunities across the entire CRM interface. A single module will handle all three sales motions:

  • New Business — fresh logo / new product sale
  • Renewal — contract renewal, AMC, SaaS subscription
  • Expansion — upsell or cross-sell to existing account

New Business Pipeline Stages

  • Pipeline Discussion
  • Scope of Work
  • Quotation Shared
  • Purchase Negotiation
  • Closed Won
  • Closed Lost

Stages are enforced via Blueprint — mandatory fields required before each transition.

Deal Value Structure — Auto-Calculated

The Opportunity Amount is calculated automatically from five line-item components:

Section 04
Renewal Management Engine
ⓘ
Core Design Principle

Renewals are NOT a separate pipeline. They are managed within the same Opportunities module using Type = Renewal, governed by a dedicated Blueprint with its own 8-stage process.

Renewal Pipeline Stages

  • Not Due
  • Upcoming
  • In Discussion
  • Quote Shared
  • PO Received
  • Payment Pending
  • Closed Won
  • Churned

Auto-Creation Triggers

Renewal Opportunities are automatically created as new records via a scheduled Deluge function:

  • 90 days before contract expiry — Stage: Upcoming
  • 60 days before contract expiry — Reminder escalation
  • 30 days before contract expiry — Urgent flag

Implemented via Zoho CRM Custom Schedule + Deluge zoho.crm.createRecord()

Renewal-Specific Fields

  • AMC Financial Year
  • AMC Month
  • AMC PO Number
  • AMC Received Date
  • Previous Contract Value
  • Renewal Type (Renewal / Upsell / Downsell / Churn)
  • Renewal Owner
  • Contract Expiry Date

Renewal Ownership Model — Recommended

We recommend a Hybrid Model: the Key Account Manager (KAM) retains primary ownership for relationship and commercial negotiation, while Customer Success (CS) is assigned as a co-owner for onboarding renewal tasks and health-check activities. The system will enforce ownership assignment via Blueprint "Before Transition" rules — a renewal cannot progress without a defined owner.

Deal Lifecycle Governance

  • Quote Validity: 60 days from creation
  • Extensions Allowed: Max 2 (each requires approval)
  • Auto Closed Lost: 180 days of inactivity
  • Extension tracking: System field counts extensions
  • Auto-close engine: Daily Deluge schedule checks deal age

Discount Thresholds — Per Component

Discounts below the threshold are auto-approved. Above the threshold, the Deal Desk approval workflow is triggered automatically.

Component Auto-Approved Up To Above Threshold
SaaS 30% Triggers Level 1 → Level 2 Approval
Professional Services 30% Triggers Level 1 → Level 2 Approval
Hardware 10% Triggers Level 1 → Level 2 Approval
Cloud Service Provider (CSP) 0% — Always Requires Approval Any discount triggers immediate approval
Other Charges 30% Triggers Level 1 → Level 2 Approval
Hardware AMC 30% Triggers Level 1 → Level 2 Approval
Software AMC 30% Triggers Level 1 → Level 2 Approval
⚠
Implementation Note — Multi-Component Discount Logic

Zoho CRM's native Approval Process supports single-field threshold conditions. To handle per-component discount thresholds across 7 components simultaneously, a Deluge custom function will evaluate all component discounts and route the approval accordingly. This is achievable but requires custom coding.

Section 06
Account Governance & Data Control

Account Creation — Strict Control

Only Df-OS Admin can create Account records. All other user profiles will have the "Create" permission disabled for the Accounts module.

Request flow for Sales team:

  • Sales submits request with: Company Name, GSTIN, Billing Address, Shipping Address, Contact details
  • Admin validates and creates the Account
  • Sales is notified and account ownership assigned

Account Hierarchy — Plants as Sub-Accounts

Zoho CRM's native Parent Account lookup field will be used to model the corporate hierarchy:

Parent Account (Corporate HQ)
  └─ Plant / Location 1
  └─ Plant / Location 2
  └─ Plant / Location N

Revenue, activities, and Opportunities roll up to the parent account automatically.

Centralized Pricing Control

A centralized Zoho CRM Price Book will be implemented and linked to all Quotes. Manual price override outside the approval system will be prevented via validation rules and Blueprint transitions. No quote can be submitted without products linked to the approved price list.

Section 08
Limitations & Recommended Additions
⚠
Important: The following requirements CANNOT be fully delivered within Zoho CRM alone

These are architectural gaps — not technical failures. Each has a clear resolution using adjacent Zoho products already in the ecosystem.

❌ MRR (Monthly Recurring Revenue) — Not Native in Zoho CRM

Zoho CRM tracks Opportunities and deal amounts — it does not understand subscription intervals, billing cycles, or recurring revenue math. There is no built-in MRR field, formula, or report in Zoho CRM. Any attempt to approximate MRR using formula fields would be manual, fragile, and inaccurate as the subscription base scales.

Recommended Resolution: Add Zoho Billing to the Scope

Zoho Billing (formerly Zoho Subscriptions) is purpose-built for SaaS subscription management. It natively tracks Active Subscriptions, Renewal Due Dates, Upgrades/Downgrades, and calculates MRR and ARR automatically. It integrates directly with Zoho CRM (subscriptions linked to Contacts/Accounts) and pushes data into Zoho Analytics for dashboard visibility. This is the correct architectural layer for MRR tracking — not CRM.

❌ ARR (Annual Recurring Revenue) — Not Native in Zoho CRM

Same root cause as MRR. ARR = MRR × 12 for monthly-billed customers, or the contracted annual value for annual subscriptions. Zoho CRM has no mechanism to distinguish one-time deal amounts from recurring contract values, making ARR calculations impossible without an external subscription ledger.

Recommended Resolution: Zoho Billing + Zoho Analytics

Zoho Billing tracks ARR natively per subscription. Zoho Analytics then combines CRM pipeline data with Billing ARR data to produce executive-level revenue dashboards showing ARR growth, contraction, and net revenue retention.

❌ Subscription Lifecycle Tracking — Limited in CRM

Tracking whether a subscription is Active, Paused, Upgraded, or Downgraded — and automatically updating status on billing events — is not a CRM function. Zoho CRM can store static custom fields (e.g., "Subscription Status") but cannot react to billing events, payment failures, or plan changes without an external trigger.

Recommended Resolution: Zoho Billing as Subscription System of Record

Zoho Billing manages the subscription state machine. When a subscription is upgraded, downgraded, or churned in Billing, a webhook or native sync updates the corresponding CRM record automatically. Custom fields in CRM will reflect the live state from Billing — no manual updates needed.

⚠ BOM Visibility in CRM — Requires Custom Integration

Zoho Inventory manages Bill of Materials (via Composite Items), but there is no native way to surface BOM details inside a CRM Opportunity view. Sales teams would need to switch to Inventory to see component breakdowns.

Resolution: Custom Related List via Zoho API

A custom Deluge function will pull BOM data from Zoho Inventory's API and display it as a read-only custom related list inside the relevant Opportunity or Quote record in CRM. This provides the required visibility without leaving the CRM interface.

⚠ Sub-Partner Hierarchy in Partner Portal — Requires Custom Module

Zoho CRM Portals natively support one level of external access (Partner sees their own deals). A multi-tier sub-partner hierarchy (Partner → Sub-Partner → End Customer) is not a built-in portal feature.

Resolution: Custom Partner Hierarchy Module

A custom module will model the partner-subpartner relationship. Portal access rules will be configured so each sub-partner only sees deals they own, while the parent partner sees their entire sub-partner tree. This requires custom configuration but is achievable within Zoho CRM.

Recommended Additional Products

Zoho Billing (Critical Addition)

Required for MRR, ARR, subscription lifecycle, upgrade/downgrade tracking. Without this, revenue metrics will be inaccurate estimates, not system-calculated values.

Zoho Analytics (Strongly Recommended)

Required for cross-system dashboards (CRM + Books + Billing data in one view). Native CRM dashboards cannot pull data across multiple Zoho apps simultaneously.
